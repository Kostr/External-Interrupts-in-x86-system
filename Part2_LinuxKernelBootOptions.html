External Interrupts in the x86 system. Part 2. Linux kernel boot options

In the last part we've discussed evolution of the interrupt delivery process from the devices in the x86 system (PIC → APIC → MSI), general theory and all the necessary terminology.

In this practical part we will look at how to roll back to the use of obsolete methods of interrupt delivery in Linux, and particulary we will look at Linux kernel boot options:

  • pci=nomsi
  • noapic
  • nolapic

Also we will look at the order in which OS looks for interrupt routing tables (ACPI/MPtable/$PIR) and what impact is from the following boot options:

  • pci=noacpi
  • acpi=noirq
  • acpi=off

Probably you've used some combination of these options, when one of devices didn't work correctly in your system because of the interrupt problem. We'll go through these options and find out what they do and how they change kernel '/proc/interrupts' interface output.

File '/proc/interrupts' is the procfs Linux interface to the interrupt subsystem and it presents a table about amount of the interrutps on every CPU core in the system in the following form:

  • First column: interrupt number
  • CPUx columns: interrupt counters for every CPU core in the system
  • Next column: interrupt type:
    • IO-APIC-edge - edge-triggered interrupt for the I/O APIC controller
    • IO-APIC-fasteoi - level-triggered interrupt for the I/O APIC controller
    • PCI-MSI-edge - MSI interrupt
    • XT-PIC-XT-PIC - interrupt for the PIC controller (we will see it later)
  • Last column: device (driver) associated with this interrupt

So all here is like it is supposed to be in the modern system. For the devices and drivers which support MSI/MSI-X, this is the type of interrupt that they use. The rest of the interrupt routing is happening through the APIC controller

nolapic

This kernel boot option disables LAPIC. MSI interrupts can't work without LAPIC, I/O APIC can't work without LAPIC. Interrupt routing tables and the options "acpi=noirq", "pci=noacpi", "acpi=off"

How does the operating system gets information about the device interrupt routing? BIOS prepares such info for the OS in the form of:

  • ACPI tables (_PIC/_PRT functions)
  • _MP_ table (MPtable)
  • $PIR table
  • Registers 0x3C/0x3D of the devices PCI configuration space

It is worth to notice that for the MSI interrupts declaration BIOS doesn't need to do anything extra (beside declaring the use of the LAPIC), all the aforementioned routing information is needed only for the APIC/PIC interrupt lines.

Tables in the list above are presented in the order of priority. Let's examine it in detail.

Let's assume the BIOS have presented all this data and we boot our OS without any extra boot options:

  • OS finds ACPI tables
  • ОS executes ACPI function "_PIC", passing it the argument stating that the boot should happen in APIC mode. Here is the function code usually saves the chosen mode in a variable (for example, let it be PICM=1)
  • To access interrupt routing info OS calls ACPI function "_PRT". In itself it checks PICM variable and returns routing for the APIC mode case

In a case when we boot with the option noapic:

  • OS finds ACPI tables
  • ОS executes ACPI function "_PIC", passing it the argument stating that the boot should happen in PIC mode. Here is the function code usually saves the chosen mode in a variable (for example, let it be PICM=0)
  • To access interrupt routing info OS calls ACPI function "_PRT". In itself it checks PICM variable and returns routing for the PIC mode case

If ACPI tables aren't present or the functionality of the interrupt routing with the help of ACPI is disabled through the option acpi=noirq or pci=noacpi (or ACPI subsystem is completely disabled with the acpi=off option), then OS looks for the MPtable (_MP_) to get all the interrupt routing information:

  • OS can't find/don't look at the ACPI tables
  • OS finds MPtable (_MP_)

If ACPI tables aren't present or the functionality of the interrupt routing with the help of ACPI is disabled through the option acpi=noirq or pci=noacpi (or ACPI subsystem is completely disabled with the acpi=off option), and if the MPtable (_MP_) is not present either (or there is a boot option noapic or nolapic):

  • OS can't find/don't look at the ACPI tables
  • OS can't find/don't look at the MPtable (_MP_)
  • OS finds $PIR table

If there is no $PIR table or it is not full, then the operation system will look at the registers 0x3C/0x3D of the devices PCI configuration space to guess interrupt routing.

Сonclusion

In a conclusion one more time go through the discussed options.

Interrupt controller choise options:

  • pci=nomsi - MSI interrupts become IO-APIC/XT-PIC depending of the interrupt controller in use
  • noapic - Disables I/O APIC. MSI interrupts can still go to all the other CPUs, the rest of the device interrupts can only go to the PIC, and it works with the CPU0 only. But LAPIC works and other CPUs can work and handle interrupts
  • noapic pci=nomsi - All of the device interrupts can only go to the PIC, and it works with the CPU0 only. But LAPIC works and other CPUs can work and handle interrupts
  • nolapic - Disables LAPIC. MSI interrupts can't work without LAPIC, I/O APIC can't work without LAPIC. All of the device interrupts can only go to the PIC, and it works with the CPU0 only. And without LAPIC the rest of the CPUs besides CPU0 wouldn't work.

Interrupt tables priority options:

  • no options - routing through the APIC with the help of ACPI tables
  • noapic - routing through the PIC with the help of ACPI tables
  • acpi=noirq (pci=noacpi/acpi=off) - routing through the APIC with the help of MPtable
  • acpi=noirq (pci=noacpi/acpi=off) noapic (nolapic) - routing through the PIC with the help of $PIR
